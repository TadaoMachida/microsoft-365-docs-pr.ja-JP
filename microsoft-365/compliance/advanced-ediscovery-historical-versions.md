# <a name="set-up-historical-versions-in-advanced-ediscovery-preview"></a>バージョン管理 (プレビュー) でAdvanced eDiscoveryバージョンを設定する

Advanced eDiscovery の履歴バージョン機能を使用すると、組織内の電子情報開示管理者は、SharePoint Online および OneDrive for Business に保存されているドキュメントのすべてのバージョンのコンテンツを検索して収集できます。 次に、そのコンテンツを分析およびレビュー用のレビュー セットに追加できます。 これにより、同じドキュメントの最新バージョンに関連する情報が含まれている場合でも、ケースや調査に関連する可能性のある特定のバージョンのドキュメントからコンテンツを検索して確認できます。

サイトのバージョン履歴機能をサポートするには、Advanced eDiscovery管理者SharePointサイトのバージョン管理を有効にする必要があります。 次に、ユーザーがドキュメントを SharePointまたは OneDriveに変更すると、ドキュメントを保存 (または自動保存) するときに暗黙的な通常のバージョンが作成されます。 SharePointバージョン管理を使用すると、アイテム (ドキュメント、イベント、タスクを含む) でSharePointアクティビティを追跡できます。 このバージョン管理機能は、法的調査で証拠を提供できる監査証跡を残します。 これらの古いバージョンのドキュメントは、法的な問題で裁判所の検出中に機密または関連するコンテンツを持つそのようなバージョンを共有する必要がある場合があります。

電子情報開示管理者が組織の履歴バージョンを有効にしてから、特定の SharePoint サイトに対してアクティブ化すると、SharePoint コンテンツ プッシュ サービスは、アクティブ化されたサイト上のすべてのメジャー バージョンとマイナー バージョンのドキュメントをクロールし、それらのバージョンをインデックス作成用に送信します。 クロールとインデックス作成プロセスが完了すると、ドキュメントとそのバージョンを電子情報開示検索で使用できます。 特定のバージョンに (バージョン履歴で) アクセスできる限り、そのバージョンはコレクションの検索Advanced eDiscoveryできます。

## <a name="set-up-historical-versions"></a>履歴バージョンを設定する

Advanced eDiscovery で履歴バージョンを有効にするには、組織で有効にして特定のサイトをアクティブ化して、それらのサイトに保存されているドキュメントのすべてのバージョンが検索用にインデックス化される必要があります。 履歴バージョンのAdvanced eDiscovery設定する前に、バージョン管理のサポートを有効にするSharePoint。

### <a name="step-1-turn-on-versioning-in-sharepoint"></a>手順 1: バージョン管理でバージョン管理を有効SharePoint

最初の手順は、SharePointオンラインでバージョン管理を有効にし、すべてのバージョンのドキュメントを保持します。 手順については、「バージョン管理」[を参照SharePoint。](/microsoft-365/community/versioning-basics-best-practices)

### <a name="step-2-turn-on-historical-versions"></a>手順 2: 履歴バージョンを有効にする

次の手順では、バージョン管理の履歴バージョンを有効Advanced eDiscovery。 組織の履歴バージョンを有効にするには、グローバル管理者または電子情報開示管理者 (電子情報開示マネージャー役割グループの電子情報開示管理者サブグループのメンバー) である必要があります。 履歴バージョンを有効にすると、組織全体に適用されます。

> [!IMPORTANT]
> 履歴バージョンを有効にした後は、パブリック プレビュー中にオフにできます。 一般提供のために履歴バージョンがリリースされた後で、この機能をオフにできます。

1. [設定] Microsoft 365 コンプライアンス センターに移動し [、[Advanced eDiscovery]](https://go.microsoft.com/fwlink/p/?linkid=2173764)をクリックAdvanced eDiscovery **します**。

   ![[設定Advanced eDiscovery選択]](..\media\HistoricalVersions1.png)

2. [バージョン **設定]** ページで、[バージョン履歴 **] (プレビュー)** タブを選択し、[履歴バージョン] テナント コントロールのトグルをオンに切り替えます。

   ![トグルをオンにして履歴バージョンを有効にする](..\media\HistoricalVersions2.png)

   履歴バージョンをオフにできないという警告が表示されます。 前に述べたように、機能が一般提供のためにリリースされるまで、過去のバージョンをオフに戻す事はできないでしょう。

3. [ **はい] を** クリックして、履歴バージョンを有効にしてください。

### <a name="step-3-activate-sharepoint-sites"></a>手順 3: サイトをSharePointする

> [!NOTE]
> 過去のバージョンのパブリック プレビューでは、組織ごとに 100 のサイトライセンス認証の制限があります。 アクティブ化は、過去のバージョンのサイトを有効または無効にするたびに、この制限に対してカウントされます。 複数のサイトを有効にした場合、各サイトは 1 つのライセンス認証としてカウントされます。 ライセンス認証の総数が [履歴バージョン] **タブに表示** されます。

1. [バージョン **管理] ページ** の [Advanced eDiscovery設定] タブで **、[** 有効]をクリックして、履歴バージョンのサイトをアクティブにします。

   ![[有効にする] をクリックして、過去のバージョンのサイトをアクティブ化する](..\media\HistoricalVersions3.png)  

   組織内のすべてのサイトの一覧を含むSharePointが表示されます。

2. アクティブにするサイトを選択し、[有効] **をクリック** して、過去のバージョンに対してアクティブ化します。 検索ボックスを使用して、特定のサイトを検索できます。

   サイト上のドキュメント のバージョンにインデックスが作成され、このインデックス作成プロセスは、バージョンを検索する準備が整うまでに時間がかかるという警告が表示されます。 また、選択したサイトの履歴バージョンを 30 日間無効にできないという警告も表示されます。

3. [ **はい] を** クリックして、過去のバージョンのサイトをアクティブ化します。

   ![アクティブ化されたサイトとサイトのライセンス認証の数が表示されます。](..\media\HistoricalVersions4.png)  

   サイトがアクティブ化されたサイトの一覧に追加されます。 サイトのアクティブ化の数を示すカウンターも更新されます。

4. 過去のバージョンに対してアクティブ化するサイトごとに、前の手順を繰り返します。

## <a name="frequently-asked-questions"></a>よく寄せられる質問

**下書きコレクションをレビュー セットにコミットするときに"すべてのバージョンを収集する" オプションと、履歴バージョンの違いは何ですか?**

現在、検索用にインデックスが作成されているのは最新バージョンのドキュメントのみです。 つまり、下書きコレクションを実行すると、最新バージョンのドキュメントだけが検索されます。 ドキュメントがコレクションのキーワード クエリと一致する場合は、コレクションの結果に返されます。 ただし、ドキュメントの最新バージョンが検索クエリと一致しない場合、古いバージョンのドキュメントにキーワードが含まれている場合、ドキュメントは返されません。 この状況を軽減するために、Advanced eDiscovery コレクションをレビュー セットにコミットするときに、ドキュメントのすべてのバージョンを収集[できます](commit-draft-collection.md#commit-a-draft-collection-to-a-review-set)。 つまり、キーワードを含む古いバージョンがレビュー セットに追加されます。

サイトをアクティブ化すると、ドキュメントのすべてのバージョン (最後のバージョンではなく) が検索用にインデックス付けされるので、履歴バージョンは"すべてのバージョンを収集する" とは異なる効率的です。 その結果、古いバージョンのドキュメントに検索クエリに一致するキーワードが含まれている場合、コレクションによって返されます。

**サイトに対して履歴バージョンが有効になっている場合、サイトのパフォーマンスに影響しますか?**

いいえ。 サイトの履歴バージョンが有効になると、サイトのパフォーマンスは、サイトが有効にされる前と同じになります。 有効にした後にサイトで実行されるクロールおよびインデックス作成プロセスは、低速で発生し、ピーク時外に実行されます。 サイトの履歴バージョンを有効にすると、バックフィル プロセスが開始され、サイト上のすべてのバージョンのドキュメントが検索され、それらのバージョンがインデックスに送信されます。 サイトのドキュメント バージョンの数によっては、このバックフィル プロセスがサービスの正常性に影響を与える可能性があります。 この潜在的な影響は、次の方法で軽減されています。

- オフピーク時にこれらのバージョンを処理するために最善の努力をします。

- 最も優先度の低いキューでドキュメント バージョンを処理します。これにより、ほとんどのサービス リソースをユーザーの変更に委任できます。

**サイトのすべての履歴バージョンのドキュメントがインデックス付けされ、電子情報開示検索に使用されるまで、サイトがアクティブ化されるまで、どのくらいの期間待つ必要がありますか?**

サイトのドキュメント数とドキュメントごとの平均バージョン数に基づいて、サイトごとのファイルの総数を推定します。 これを基に、インデックス作成にかかる時間の見積もりは次のとおりです。

`Number of versions / (Processing rate of 100,000 files per day ) + .5 days = Total number of days to process a site`

サイト上のバージョンのインデックス作成はオフピーク時に実行するために最適化されたので、半日はバッファーとして追加されます。

たとえば、サイトのドキュメントとすべてのバージョンの総数が 150,000 の場合、サイトの履歴バージョンの処理には少なくとも 2 日かかる場合があります。

`150,000 files/100,000 files per day + .5 days = 2 days`

**履歴バージョンのサイトを頻繁にアクティブ化または非アクティブ化しない理由**

以前にアクティブ化されたサイトを非アクティブ化すると、クリーンアップ プロセスがトリガーされます。 このプロセスの完了には時間が必要です。 同じサイトが再度アクティブ化された場合は、サイトのインデックスを再作成するバックフィル プロセスを再実行する必要があります。 クリーンアップ プロセスとバックフィル プロセスの両方が時間とリソースを消費します。 したがって、サイトの履歴バージョンを繰り返しアクティブ化および非アクティブ化しないように、履歴バージョンでアクティブ化するサイトを慎重に検討し、計画してください。
